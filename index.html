História do HTML

Voltar

HTML surgiu no início da década de 1990, criado por Tim Berners-Lee, cientista britânico que também inventou a World Wide Web. A primeira versão do HTML foi extremamente simples, com poucas tags e sem suporte para estilos ou interatividade.

Ao longo dos anos, o HTML evoluiu significativamente:
  • HTML 2.0 (1995): Padronizou os primeiros elementos usados na web.

  • HTML 3.2 (1997): Introduziu suporte a scripts e estilos.

  • HTML 4.01 (1999): Tornou-se mais robusto, com foco em acessibilidade e estrutura.

  • HTML5 (2014): Revolucionou a web com suporte nativo a vídeo, áudio, gráficos, semântica e APIs modernas.
Hoje, o HTML é a base da construção de páginas web, sendo constantemente atualizado para atender às novas demandas da internet moderna.

Cliente x Servidor

Voltar

O modelo cliente-servidor é a base da comunicação na web.

Ao acessar um site o navegador envia uma requisição ao servidor, que responde com os arquivos HTML, CSS e JavaScript necessários para exibir a página.

Este modelo permite que múltiplos clientes acessem simultaneamente os recursos de um servidor, tornando a web escalável e eficiente.

Cliente

Sempre que um dispositivo com acesso à internet visita um site, o navegador consulta a estrutura da página e realiza o download dos arquivos necessários — como HTML, CSS, JavaScript e imagens. Esses arquivos são armazenados temporariamente no cache do navegador, permitindo que futuras visitas ao mesmo site sejam mais eficientes.

Após o navegador copiar os arquivos do site e armazená-los em cache, ele carrega a estrutura da página localmente. Isso permite que o conteúdo seja exibido de forma rápida e eficiente, tornando a página acessível ao usuário com menor tempo de espera.

Servidor

No lado do servidor, quando uma requisição HTTP é recebida, ela passa por uma série de etapas até ser processada. O servidor identifica qual recurso foi solicitado como uma página HTML, um arquivo CSS, um script JavaScript ou até dados de uma API e determina como responder com base nas rotas e regras configuradas na aplicação.

Se o conteúdo solicitado for estático (como imagens ou arquivos HTML prontos), o servidor simplesmente localiza e envia o arquivo correspondente. Já em casos de conteúdo dinâmico, o servidor pode precisar executar scripts, acessar bancos de dados, processar lógica de negócio ou interagir com outras APIs antes de gerar uma resposta.

Uma vez que os dados ou arquivos apropriados são preparados, o servidor os encapsula em uma resposta HTTP e os envia de volta ao cliente, acompanhados de cabeçalhos que definem detalhes importantes, como tipo de conteúdo, status da requisição e políticas de cache.

Para que seu site apareça para o mundo, ele precisa de dois elementos essenciais: um servidor de hospedagem e um domínio próprio. No Brasil, o registro desse domínio é feito pelo Registro.BR, é ele que garante um endereço único na internet.

  3. head: cabeçalho onde ficar os metadados do documento, como metatgs, css e javascript.
  4. meta: metatags, como charset, description, viewport ou keywords
  5. title: define o título do documento, mostrado na barra de títulos do navegador e como link nos mecanismos de busca.
  6. body: corpo ou área principal do documento, dentro deste deve ficar o conteúdo da sua página.

Acessibilidade

Voltar

WAI-ARIA, o Accessible Rich Internet Applications Suite, define uma maneira de tornar o conteúdo e os aplicativos da web mais acessíveis a pessoas com deficiência. Ele ajuda especialmente com conteúdo dinâmico e controles avançados de interface do usuário desenvolvidos com HTML, JavaScript e tecnologias relacionadas.

Aprender HTML, CSS e Javascript é útil se você quer se tornar um desenvolvedor web, mas seu conhecimento precisa ir além. Você deve usá-las de forma a maximizar a audiencia dos seus sites, e permitir que todos possam acessar o conteúdo. Para conseguir isso, será necessário seguir boas práticas, realizar testes em vários navegadores e pensar em acessibilidade desde o início do desenvolvimento.

Sem o WAI-ARIA, certas funcionalidades usadas em sites, não estão disponíveis para alguns usuários com deficiência - especialmente pessoas que dependem de leitores de tela e pessoas que não podem usar um mouse. O WAI-ARIA aborda esses desafios de acessibilidade, por exemplo, definindo maneiras de fornecer funcionalidade à tecnologia assistiva. Com o WAI-ARIA, os desenvolvedores podem tornar os aplicativos da Web avançados acessíveis e utilizáveis para pessoas com deficiência.

Semântica

Voltar

A semântica em HTML se refere ao uso de tags que têm significado próprio, ou seja, que indicam claramente o papel daquele conteúdo dentro da estrutura de uma página. Em vez de apenas dizer "aqui tem algo", o HTML semântico diz "aqui tem um cabeçalho", "aqui tem um artigo", ou "aqui tem uma navegação".

A semântica é importante por conta da acessibilidade, SEO, Manutenção e legibilidade, entre outros. Leitores de tela e outras tecnologias assistivas conseguem interpretar melhor o conteúdo, assim como, motores de busca entendem com mais clareza a estrutura e o propósito de cada parte do conteúdo. O código fica mais organizado, fácil de entender e manter, possibilitando que aplicações e ferramentas externas possam interagir melhor com o site.

  3. main: Conteúdo principal da página
  4. section: Seções temáticas dentro do conteúdo
  5. article: Conteúdo independente, como postagens ou notícias
  6. aside: Conteúdo complementar (como barras laterais).
  7. footer: Rodapé da página ou de uma seção

Utilizar HTML semântico é mais do que uma boa prática, é uma forma de tornar a web mais acessível, compreensível e eficiente para todos. Ao aplicar a semântica corretamente, você contribui para uma internet mais inclusiva e melhora a qualidade técnica do seu projeto.
